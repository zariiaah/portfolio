About

Most of my projects are an implementation of where Roblox meets everything else - moderation pipelines, announcements, staff commands, HTTP bridges and data that has to match between Discord and the game.

My main priorities are clear ownership (what runs on the server, what sits in a bot, what you expose over HTTP), sensible rate limits and logging your team can act on.

Outside of Roblox I build small APIs, static or lightly dynamic pages and the glue code that keeps those pieces in sync with your experience.

For ongoing products I am happy owning a QA mindset; scripted or exploratory checks, reproducing player or staff reports, regression passes after a fix and written notes your team can file or paste into tickets. Bug fixing is often where I start when a codebase is new to me. For greenfield work I like a thin spec first, then milestones so a complex idea does not collapse halfway through integration.

Signed admin relay

Staff trigger a Discord action that posts a signed payload to your game over HttpService. The server checks the secret, enforces a simple rate limit and runs a fixed list of actions (kick, config flip, shard message).

Account link and roles

Players verify a Roblox account to the bot; you store the mapping, mirror Discord roles if needed and read the same data in-game for gates or staff lists.

Python or Node

Metrics digest to Discord

Your game already writes numbers (for example concurrent players or custom counters). A small script on a schedule reads the latest values and posts one Discord message when something notable happens, like a sudden drop in players or a count passing a limit you set.

Web · Node

Feature flag panel

Turn features on or off without publishing a new Roblox build: a tiny website or API holds on/off switches, the game asks for them when it starts or every few minutes and you can lock changes to staff on Discord if you want.
